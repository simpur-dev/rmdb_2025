# Buffer Pool 缓存的核心页类型

## 1. 数据页（Data Page）

**最简单理解：存真实业务数据的页**

- 比如你建了一张 `user` 表，里面的 `id`、`name`、`age` 这些一行行记录，就存在数据页里。
- 大小默认 **16KB**，是数据库最基础的存储单元。
- 你执行 `select * from user where id=1`，加载到内存的就是数据页。

## 2. 索引页（Index Page）

**理解：存 B+ 树索引结构的页**

- 主键索引、普通二级索引，都是 B+ 树。
- B+ 树的每一个节点，就是一个索引页。
- **作用：** 快速定位数据在哪，不用全表扫描。
- 你写的 B+ 树索引，每个节点就是索引页。

## 3. undo 页（Undo Page）

**理解：存"修改前旧数据"的日志页，用来回滚和多版本**

你执行 `update user set name="xxx"` 时，InnoDB 会把修改之前的 `name` 存到 undo 页。

**两大作用：**

1. 事务报错了，可以**回滚**
2. 实现 **MVCC（多版本并发控制）**，让别的事务能读到旧版本数据

它也是一页一页存在磁盘，会被加载进 Buffer Pool。

## 4. 插入缓冲 / Insert Buffer（现在叫 Change Buffer）

**理解：专门给二级索引用的"写合并优化区"**

- 二级索引（非主键索引）是无序的，每次插入都要随机 IO，很慢。
- InnoDB 就把这些插入先暂存在 Insert Buffer 里，等之后空闲了，再批量合并写到磁盘索引页。
- **目的：** 把大量随机 IO → 批量顺序 IO，大幅提升写入性能。

## 5. 自适应哈希索引 / Adaptive Hash Index (AHI)

**理解：InnoDB 自动帮你建的"高速查询索引"**

- B+ 树查找要从根到叶子，多几层就慢一点。
- InnoDB 监控哪些索引页被频繁查，自动在内存里建一个哈希表，让等值查询 `where id=?` 直接 **O(1)** 找到。
- 完全自动，不用你管，只存在内存（Buffer Pool）里。

## 6. 锁信息（Lock Info）

**理解：存锁的元数据，不是真正的锁**

- 比如行锁、表锁、gap 锁的信息：锁住了哪一行、哪个事务加的、加的什么锁。
- 这些信息会在内存里维护，一部分结构放在 Buffer Pool 相关区域。
- 你代码里的 `latch_` 是内存闩锁，保护结构；这里的锁信息是**业务事务锁**。

---

## 总结（对应你的缓冲池）

Buffer Pool 不只是缓存表数据，还会缓存数据库运行需要的所有核心页/结构：

| 类型 | 说明 |
|------|------|
| **数据页** | 真正的业务行数据 |
| **索引页** | B+ 树索引节点 |
| **undo 页** | 回滚 + MVCC 旧数据 |
| **Change Buffer** | 二级索引写优化 |
| **自适应哈希索引** | 内存自动加速索引 |
| **锁信息** | 事务锁的元数据 |

> 你自己写 RMDB 时，至少要实现前两个：**数据页 + 索引页**。

# 主键 & 非主键 / 主键索引 & 二级索引 核心知识点总结

## 一、主键（Primary Key）

**定义：** 表中用于唯一标识一行数据的字段，是数据的"身份证"

**硬性规则：**

- 必须**唯一**，不可重复
- 必须**非空**，不能为 NULL
- 一张表只能有**一个**主键

**作用：** 作为数据库定位数据的核心标识，自动构建主键索引

## 二、主键以外的键（非主键列）

分为两类，均不具备主键的强制约束：

### 普通列 / 普通索引键

- 无唯一性限制，值可以重复（如姓名、年龄、性别）
- 可建普通二级索引，也可不建索引

### 唯一键（UNIQUE）

- 手动设定为唯一，但**允许 NULL**（仅这一点和主键不同）
- 会构建唯一二级索引

## 三、主键索引（聚簇索引）

**本质：** 以主键为键构建的 B+ 树索引，一张表仅有**一个**

**结构特点：**

- **非叶子节点：** 存储主键值，用于查找
- **叶子节点：** 直接存储一整行**完整数据**

**性能特点：**

- 主键通常自增，插入位置始终在链表尾部，属于**顺序 IO**，速度极快

**查询特点：** 通过主键查询，直接拿到整行数据，**无需二次查找**

## 四、二级索引（辅助索引）

**本质：** 以非主键列为键构建的 B+ 树索引，一张表可以有**多个**

**结构特点：**

- **非叶子节点：** 存储索引列的值（如姓名、年龄）
- **叶子节点：** 仅存储「索引值 + 对应主键」，**不存完整行数据**

**性能特点：**

- 索引列值无序、插入位置随机，属于**随机 IO**，写入速度慢
- 数据库用 **Change Buffer** 合并随机写，优化性能

**查询特点：** 先查二级索引拿到主键，再通过主键索引查完整数据，这个过程叫**回表**

## 五、核心对比（一张表看懂）

| 对比维度 | 主键索引 | 二级索引 |
|----------|----------|----------|
| 数量 | 仅 1 个 | 可多个 |
| 构建依据 | 主键 | 普通列 / 唯一键 |
| 叶子节点内容 | 整行完整数据 | 索引值 + 主键 |
| 查询方式 | 直接查全量数据 | 需**回表**查主键索引 |
| 插入 IO 类型 | 顺序 IO（快） | 随机 IO（慢） |
| 唯一性约束 | 强制唯一 + 非空 | 普通索引可重复，唯一索引仅唯一 |

## 六、关键补充总结

- **主键**是逻辑唯一标识，**主键索引**是物理存储结构
- 二级索引慢的核心原因：**随机 IO**，而非索引本身结构问题
- **回表**是二级索引的天然特性，也是其查询比主键索引慢的原因
- 普通非主键列/索引无需唯一，只有主键、唯一键要求唯一

多 Instance 知识点
作用：将一个大 Buffer Pool 拆成多个独立小缓冲池，减少高并发下的锁竞争。
结构：每个 Instance 拥有独立的 Free、LRU、Flush 链表和独立锁。
优势：不同线程操作不同 Instance，互不干扰，并发性能大幅提升。
参数：innodb_buffer_pool_instances，控制实例数量，最大 64 个。
意义：解决全局锁争抢问题，是高并发场景下缓冲池的核心优化。
Chunk 块知识点
作用：Instance 的最小内存管理单元，用于在线动态调整 Buffer Pool 大小。
默认大小：128MB。
结构：每个 Chunk 包含一组连续的控制块 + 缓存页。
优势：扩容 / 缩容只需增减 Chunk，无需重启数据库。
约束：buffer_pool_size 必须是 chunk_size * instances 的整数倍。
意义：实现缓冲池内存弹性伸缩，不影响业务运行。
